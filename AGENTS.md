## Concurrent tool dispatch (issue #65)

`ToolExecutor` (`engine/tool_execution/__init__.py`) may **overlap** the tool calls in one assistant message, but only within strict safety rails — read them before touching this path:

- **Only provably read-only-per-call calls overlap**, and only with each other. `concurrency.is_parallel_safe` is the single gate and is deliberately conservative: a false negative just misses a speed-up, a false positive can corrupt shared state. `read_file`/`fetch_web` qualify for any args; `bash` only when `is_read_only_command` proves it (allow-listed `|`-pipeline, no `> < ; & $( \`` metachars). `write_file`, `read_tool_artifact` (makes its own LLM call), `delegate` (a subagent can do any CRUD mid-run — deferred), and unknown tools never qualify. **Do not add a tool to the allow-list without proving it cannot write local or remote state under any argument set** (e.g. `sort -o`, `uniq out`, `tree -o`, `env VAR=x cmd`, `curl -o`/`-X POST` are all write-capable and excluded on purpose).
- **A non-overlappable call is a barrier** (`_plan_groups`): ordering across groups is preserved exactly like the old sequential path, so results always map to their `tool_call_id` regardless of finish order.
- **The shared mutable process state must never be touched concurrently.** Only the tool *invocation* runs on worker threads; arg prep, artifact-store writes/dedup, envelope building, result assembly, and every recorder emit stay on the main thread in call order. The `Recorder`'s `before_tool`/`after_tool` share a single `_pending` slot (a strict before→after pairing) — the concurrent path skips `before_tool` and passes `after_tool` an explicit `timing=(start_mono, end_mono, started_at)` bracket instead, leaving `_pending`/`_pending_child_llm_calls` untouched. If you make more of the dispatch path concurrent, keep these off the worker threads or the trace/session files will corrupt.
- Bounded by `MCA_TOOL_MAX_CONCURRENCY` (default 4; `1` disables overlap). Stdlib only (`concurrent.futures.ThreadPoolExecutor`, `contextvars.copy_context`).

## Skill knowledge delivery (issue #19)

Skills (`engine/tool_registry/skills.py`) inject user-authored procedural knowledge without editing source. Sharp edges to preserve:

- **The index goes on the opening user message, never the system prompt.** This is load-bearing for the #75 prefix-cache invariant — the system prompt must stay byte-identical before its trailing timestamp. `AgentNode._place_skill_index` appends the block to the first `role == "user"` message at construction; `_system_prompt()` takes no skill input. Do not move skill text into the system prompt.
- **`use_skill` is registered conditionally.** It is a real `ToolRegistry` method but excluded from `simple.py::_all_tools` (via `_SPECIAL_TOOLS`) and only appended by `_build_tools` when ≥1 skill is discovered — so a skill-free run's tool schemas are byte-identical to before. It also sits in `routing._BASELINE_TOOLS`, but the baseline is intersected with the real toolset, so it is only ever *effective* when actually registered.
- **The loaded-set is one mutable `set` shared by reference** from `RunContext.loaded_skills` into every per-message `ToolRegistry` (via `ToolExecutor`). That reference-sharing is what makes `use_skill` dedup persist across steps — a per-message copy would break it. On a context reset the continuation gets a *copy* of the set (seed) plus the loaded skills' full bodies re-injected after the index (`build_opening_block(skills, loaded_names)`); `delegate` passes the parent snapshot but **not** the loaded-set (the child loads independently).
- **`use_skill` is deliberately NOT in `concurrency.is_parallel_safe`** — it mutates the loaded-set, so it must never overlap. There is a regression test asserting this; keep it.
- Budget/rendering is deterministic (no LLM, no ranking): caps in `tool_execution/schema.py` (`SKILL_INDEX_*`), two degradation tiers (even truncation → names-only), never exceeding the total cap. Frontmatter is split by a hand-rolled parser — no YAML dependency; malformed frontmatter skips one skill with a warning, never fails the run.
- **Bundled example skills live at `.my_coding_agent/skills/{commit-and-push,run-quality-checks,debug-failing-run}/SKILL.md`** and ship with the repo. `.gitignore` ignores all of `.my_coding_agent/` *except* that subtree — the negation only works because the parent is kept walkable (`.my_coding_agent/*`, not `.my_coding_agent/`) before the `!.my_coding_agent/skills/` / `!.my_coding_agent/skills/**` re-includes. If you re-tighten the ignore to `.my_coding_agent/` (trailing slash, whole dir), git stops descending and the skills silently become untracked. Verify any change with `git add -n .my_coding_agent/` — it must stage *only* the three `SKILL.md` files.

## Run resilience & resume (issue #104)

The live-run and dead-run halves of one problem: survive a transient LLM failure, and continue a dead run from exactly where it stopped. Sharp edges before touching this path:

- **Classification lives only in `LLM.chat_completion` (`engine/llm/`), the single choke point every call kind funnels through** — so subagents/summarizers inherit resilience for free. `_validate_response` raises exactly one classified error (`engine/llm/errors.py`): *transport*, *http-status* (non-2xx; 5xx/429 retryable, other 4xx not), *malformed-body* (non-JSON, **or** JSON with missing/empty `choices`). A failed response must never become a successful completion or an empty assistant turn — on failure `llm_calls` is not appended and no `llm_call` event is emitted; the retry/recovery lives in `_post_chat_with_recovery` (fast transport retries, then capped-backoff patient wait bounded by `MCA_LLM_OUTAGE_TOLERANCE_S`, default 300s). Do **not** add a pipeline-level retry node.
- **`empty choices` is now a hard failure.** Historically a `{"choices": []}` response was tolerated; it is now `malformed-body`. Tests that mock `_request_with_retry` must return at least one choice, and a non-JSON-body test must use a **2xx** status (status is checked before the body, so a 500 with a non-JSON body classifies as http-status, not malformed).
- **The checkpoint is engine-owned and separate from `events.jsonl`** (observability stays passive — CONTRIBUTE §25; execution never reads the event stream back). `engine/checkpoint.py` writes `checkpoint.json` atomically (write-temp + `os.replace`) via the pipeline's `checkpoint_fn` **after each completed step**; a step that raises never reaches it, so a partial step is discarded (D4 — accept the tool-side-effect caveat, no step-level journaling).
- **A resumed run is a NEW session** (`AgentNode.from_checkpoint`): fresh id, clean trace, seeded conversation + step counter (first call is step N+1), `resumed_from` link on `session_start` and in `session_data.json` (mirrors the delegation `parent_session_id` link). Never append to the dead session's files — they stay byte-identical/auditable. CLI `--resume <id>` / `--resume-last` (`pipeline/examples/simple.py`); an unrecoverable failure sets `stop_reason = llm_failure_<classification>`, stashes `AgentNode.failure_error`, and the CLI exits non-zero with a one-line resume hint (no traceback).
- The reader/sum-check needed **no** change: `viewer/reader.py` skips unknown event types (`llm_wait`/`llm_recovery`/`llm_failure`) and a resumed session's own arithmetic sum-checks like any other.
