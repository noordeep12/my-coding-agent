# Architecture

## Overview

`my-coding-agent` is a hand-rolled Python agent harness. There are no external agent frameworks ‚Äî the entire agentic loop, tool dispatch, context management, and session persistence are implemented from scratch.

## Core Layers

### `engine/` ‚Äî Execution Primitives

The engine package owns execution primitives: the LLM HTTP client, tool dispatch, and tool definitions. The top-level `AgentNode` that drives the agentic loop lives in `pipeline/nodes/agent.py` (see below), co-located with the pipeline it builds and drives; `engine/__init__.py` re-exports `AgentNode` (lazily, to avoid a load cycle) so existing `from my_coding_agent.engine import AgentNode`-style imports keep resolving.

### `LLM` (`engine/llm/`)

The pure HTTP client. Owns the `httpx` session, calls `/v1/chat/completions`, and tracks every call in `self.llm_calls`. Construction performs no network I/O ‚Äî the model's context window is probed lazily on first access to `context_window`. Key responsibilities:

- **`chat_completion(messages, tools, kind)`** ‚Äî single POST to the LLM server; records token usage per call tagged by `kind` (`main`, `handoff`, `report`, `tool_router`, `tool_arg_correction`, `artifact_query`). This is the **single choke point** for every call kind, so failure classification and outage recovery here protect subagents and summarizers for free.
- **`available_models` / `context_window`** ‚Äî fetch the model list and resolve/cache the context window (128k fallback when unreachable).
- **`_request_with_retry`** ‚Äî retries transient connection/timeout failures with backoff (the fast phase).
- **Failure classification & outage recovery (run-resilience D1/D2)** ‚Äî `_validate_response` classifies every completion as exactly one of *transport* (`LLMTransportError`), *http-status* (`LLMHTTPStatusError`, non-2xx), or *malformed-body* (`LLMMalformedBodyError` ‚Äî non-JSON, or JSON with missing/empty `choices`); a failed response never becomes a successful completion or an empty assistant turn (`llm_calls` is not appended, no `llm_call` event is emitted). `_post_chat_with_recovery` wraps the POST in a two-phase loop: the fast transport retries, then ‚Äî for retryable classes (transport, HTTP 5xx/429) ‚Äî a **patient bounded wait** (capped exponential backoff) that keeps probing until the server answers or `MCA_LLM_OUTAGE_TOLERANCE_S` (default 300s) is exhausted. Non-retryable classes (other 4xx) fail immediately without consuming the window. Each wait/recovery/unrecoverable-failure is logged (WARNING/ERROR) and recorded as an additive `llm_wait`/`llm_recovery`/`llm_failure` event. Classified error types live in `engine/llm/errors.py`; `LLMMalformedBodyError` subclasses the pre-existing `APIResponseError` for backward compatibility.

### Tool routing ‚Äî removed (issue #204, was retained-but-unwired since #114)

`ToolRouter` (`engine/routing.py`) and `ToolRoutingNode` (`pipeline/nodes/tool_routing.py`) have been deleted. As of #114, every step already offered the full `ctx.all_tools` to the LLM instead of a routed subset, so the offered toolset is stable and predictable across steps (fixing missing-tool improvisation, non-deterministic runs, and prefix-cache churn from a per-step-changing tool list, at the cost of more schema tokens per call); the routing code stayed importable-but-unwired for a time in case the behavior needed reintroducing, then was removed outright as dead weight. Live runs are unaffected ‚Äî nothing called it. The `router`/`tool_router` event types and the reader's `_build_router_node` stay, so archived sessions recorded before #114 keep rendering their router nodes unchanged; new sessions simply emit none. The prior implementation remains available via git history if the behavior is ever needed again.

### `ToolExecutor` (`engine/tool_execution/` package)

Constructed **per assistant message** (`ToolExecutor(message, llm, tools=ctx.all_tools)`). Runs `before_tool_call` ‚Üí `invoke_tool` ‚Üí `after_tool_call` per call. Returns tool messages and records. Normalizes all results into the canonical `{schema_version, tool, ok, output, error, metadata}` envelope. For a `bash` result each datum maps to exactly one field ‚Äî `stdout`‚Üí`output`, `stderr`‚Üí`error` (whenever non-empty, **regardless of `ok`**, so a masked pipeline failure surfaces as `ok:true` with a non-null `error`), `exit_code`‚Üí`metadata.exit_code`, success‚Üí`ok` ‚Äî with nothing duplicated (`ok` is the sole success verdict; a non-null `error` does not imply failure). `result_envelope` only handles the single live artifact shape (a dict artifact carrying a preview) ‚Äî no legacy fallbacks for shapes no tool produces. The bash-style `ok`/`stdout`/`stderr`/`exit_code` sniff is gated by tool name (`bash` only), not by content shape, so a verbatim fetched JSON body that happens to contain an `ok` key is never reinterpreted as that contract. A structured-return tool's own `metadata` bag (e.g. `fetch_web`'s `{content_type, transform}`) is merged into the envelope's `metadata` untouched; `metadata.artifact`/`tool_call_id` are set only when a stream was actually offloaded (non-empty `preview`), so a small structured return is never mislabeled as offloaded. Every envelope also carries an advisory `metadata.lang = {command, output, error}` language hint (`lang.resolve_lang`, values `python`/`shell`/`json`/`text`) resolved from the tool name and args ‚Äî `bash` command‚Üí`shell`, a `read_file` output‚Üíits file extension, else a JSON sniff, else `text` ‚Äî so the trace viewer highlights each box without re-inferring; it never changes `output`/`error`/`ok`/`exit_code`. It makes no LLM calls itself ‚Äî the LLM client is held only for the session log path and the observability recorder. Forwards the run's toolset **and its `llm` client** to the `ToolRegistry` so toolset-aware tools (notably `delegate`) and `read_tool_artifact`'s extraction call can use them.

**Dangerous-command refusal gate (issue #124).** `invoke_tool` evaluates every `bash` call against `tool_execution.policy.evaluate` ‚Äî a deterministic, local, stdlib-only rule set (`policy.py`, no LLM call, no network access) ‚Äî **before** dispatch; a match short-circuits to a `reason: "refused"` no-execution failure descriptor, a fourth sibling of `not_found`/`wrong_args`/`raised`, so the command never reaches `subprocess.run`. Because `invoke_tool` is the single choke point both the sequential and the concurrent `_invoke_timed` worker path funnel through, the gate covers both by construction, and a delegated subagent inherits it for free since it constructs the identical `ToolExecutor`. Each rule (recursive root/home-path delete, remote-content-piped-to-shell, raw-device write, fork bomb, system-path permission blast, credential-file exfiltration, destructive git force-push) carries a `rule_id`, a reason, at least one CWE/OWASP/NIST reference (with URL), and a safer alternative; the rule set is deliberately narrow and high-signal (opposite bias of `is_parallel_safe`) ‚Äî a false positive blocks legitimate work, so only unambiguous danger fires. `after_tool_call` turns a refusal into the standard `ok:false` envelope: `error` is model-facing prose naming the command/reason/reference/alternative so the agent can steer, and `metadata.refusal` carries the same facts as structured fields, machine-distinguishable from `raised`/`wrong_args`/`not_found` without parsing prose. It also emits `Recorder.record_refusal` (a passive, additive `refusal` event ‚Äî the gate itself stays recorder-free) and a WARNING log line; a refusal never aborts the run ‚Äî it is one failed tool call the pipeline continues past like any other. Because this gate matches literal command text, obfuscation (base64, `${IFS}`, `eval`) can evade it by design ‚Äî an inherent gap, not one more rules can close ‚Äî so `metadata.refusal` also carries `posture_note` (issue #130): a fixed sentence stating the gate is a first layer, not a complete boundary, naming OS-level enforcement (the #25 sandbox) as what covers the rest. `policy.evaluate` checks `policy.is_gate_disabled()` first ‚Äî an opt-out via `MCA_DISABLE_DANGEROUS_COMMAND_GATE` (or the CLI's `--no-safety-gate`, which just sets that var before the run starts, printing a loud stderr warning); reading the env var at call time rather than import time means whichever set it wins for the process, no plumbing through `AgentNode`/`ToolExecutor`. See `SECURITY.md` for disabling and rule-extension guidance.

**Network egress filter (issue #126).** `invoke_tool` evaluates every `fetch_web` call against `engine.egress.evaluate` ‚Äî a deterministic, local-except-for-the-periodic-refresh check (`engine/egress/`) ‚Äî **before** dispatch, right after the refusal gate; a match short-circuits to a `reason: "egress_blocked"` no-execution failure descriptor, a fifth sibling of `refused`/`not_found`/`wrong_args`/`raised`, so the call never reaches `httpx`. `egress.is_blocked(host)` matches the destination's host (and its parent domains ‚Äî `a.b.evil.example` matches an `evil.example` entry) against a locally-cached blocklist (`engine/egress/schema.py`'s `EgressConfig`: primary source hagezi Threat-Intelligence-Feeds, secondary abuse.ch URLhaus), **deny-known-bad / allow-unknown** ‚Äî an unlisted host is never blocked. The cache (default `~/.my_coding_agent/egress_cache/blocklist.txt`) refreshes once stale (default cadence 24h); a failed refresh falls back to the last-good cache, and an absent cache with a failed refresh runs the filter *open* (a logged warning) rather than blocking all outbound work ‚Äî offline usability wins over strict enforcement here, the deliberate asymmetry against the refusal gate's local, always-available rule set. `after_tool_call` turns a block into the standard `ok:false` envelope: `error` is model-facing prose naming the host and reason so the agent can steer, and `metadata.egress` carries `{host, matched_list, reason}` as structured fields. It also emits `Recorder.record_egress` (a passive, additive `egress` event ‚Äî the check itself stays recorder-free) and a WARNING log line; a block never aborts the run. `egress.schema.is_filter_disabled()`/`EgressConfig.from_env()` check `MCA_DISABLE_EGRESS_FILTER` first (or the CLI's `--no-egress-filter`, which sets that var before the run starts, printing a loud stderr warning), read at call time like the refusal gate's opt-out. See `SECURITY.md` for disabling and source-selection guidance.

**Untrusted-content confinement (issue #128, `engine/provenance/`).** The agent ingests external content ‚Äî currently `fetch_web` results ‚Äî and without confinement treats the words in it with the same authority as the developer's task. `fetch_web` tags its returned artifact's `metadata` with `provenance.mark_untrusted()` at the point it's built (`tool_registry/registry.py`), so the marker rides through `result_envelope`'s existing `metadata` merge untouched. In `after_tool_call`'s success path, `_apply_provenance` checks `env["metadata"]["provenance"]`: when it's `provenance.UNTRUSTED`, it flips a run-scoped `contextvars` flag (`note_untrusted_content` ‚Üí `is_untrusted_active()`, never cleared once set) and wraps `env["output"]` in explicit data-not-instructions framing (`provenance.demarcate`) ‚Äî applied at this content-insertion point only, never the system prompt or tool schemas, so the #75 prefix-cache invariant holds regardless of ingestion. `_apply_provenance` also updates a second run-scoped flag from every completed `bash` call: `note_bash_command` sets `is_freshly_cloned()` the first time a successful `git clone` is observed. `invoke_tool` checks `provenance.check_reduction(func_name, args)` right after the dangerous-command gate (before dispatch, same choke point): once both flags are set, a `bash` command matching the build/install/setup pattern (`npm install`, `make`, `sh`/`bash`/`./*.sh`, `./configure`, `cargo build`, `python setup.py` ‚Äî the concrete clone-and-build threat behind the 0DIN incident referenced in the design) short-circuits to a `reason: "reduced"` no-execution descriptor, a sibling of `refused`/`not_found`/`wrong_args`/`raised`; no sandbox (#25) exists yet to route through, so this first increment refuses outright. `_build_reduction_result` turns it into the same `ok:false` envelope shape as a policy refusal (`metadata.reduction`: `rule_id`/`reason`/`safer_alternative`) so the model can steer, and emits `Recorder.record_provenance` (kind `mark` or `reduction_refusal`) ‚Äî a passive, additive `provenance` event, never echoing the ingested content itself. A run that fetches and clones nothing never sets either flag, so `check_reduction` always returns `None` and the run is byte-identical to before this change.

**Secret-exfiltration guard (issue #127).** `invoke_tool` evaluates every egress-tool call (currently `fetch_web`) against `engine.exfil.evaluate` ‚Äî a deterministic, local, stdlib-only sensitivity check (`engine/exfil/`, no LLM call) ‚Äî at the same pre-dispatch choke point the dangerous-command gate uses, checked first; a match short-circuits to a `reason: "exfil_blocked"` no-execution failure descriptor before the command ever reaches `httpx.get`. `is_sensitive(path)` matches well-known secret filename/path patterns (`.env`, `.ssh/id_*`, `.aws/credentials`, `.netrc`, `*.pem`/`*.key`/`*.p12`, `*secret*`/`*credential*` ‚Äî aligned with the project's own never-commit list) and `scan_payload(data)` additionally matches content signatures (PEM private-key headers, common token formats); `fetch_web` is GET-only today (`url` + `timeout`, no body/params), so `evaluate` scans the outbound `url` string itself ‚Äî a future POST-capable tool or `bash`/#25 subprocess-egress integration adds its own args to `exfil._EGRESS_TOOLS` the same way. Neither function ever returns the matched value, only its category name, so a block can never echo the secret back through an error message or event row. `after_tool_call`'s `_build_exfil_result` turns a block into the standard `ok:false` envelope (`metadata.exfil = {tool_name, category}`) and emits `Recorder.record_exfil` (a passive, additive `exfil` event) plus a WARNING log line; a block never aborts the run, same as a policy refusal. `exfil.evaluate` checks `exfil.is_guard_disabled()` first ‚Äî an opt-out via `MCA_DISABLE_EXFIL_GUARD` (or the CLI's `--no-exfil-guard`, mirroring `--no-safety-gate`'s pattern); disabled, egress behavior is byte-identical to before with no guard events.

**Lifecycle hooks (issue #129).** `engine/hooks/` (a `schema.py` + registry package, stdlib-only, no cycle) generalizes the refusal gate's veto shape into a registrable seam: four deterministic events ‚Äî `PreToolUse`, `PostToolUse`, `SessionStart`, `SessionEnd` ‚Äî fired by the engine, independent of model output, at developer-configured in-process Python callables (`Hooks.load()` reads `MCA_HOOKS_CONFIG`, a JSON file of `{name, event, tool, callable}` entries; unset means zero hooks, byte-identical to before this seam existed). `ToolExecutor.invoke_tool` fires every matching `PreToolUse` hook first, ahead of every other gate (`policy.evaluate`, `egress`, `exfil`, `provenance`); a hook's block decision short-circuits to a `reason: "blocked_by_hook"` no-execution descriptor, one more sibling of `refused`/`exfil_blocked`/`egress_blocked`/`reduced`/`not_found`/`wrong_args`/`raised`, so the command never reaches dispatch ‚Äî same choke point, same construction-time coverage of both dispatch paths and subagents as the refusal gate. `after_tool_call` turns a block into the standard `ok:false` envelope (`error` model-facing prose, `metadata.hook_block` structured facts) and fires `PostToolUse` on its ordered main-thread path; `AgentNode.execute` fires `SessionStart`/`SessionEnd` once each inside its own session bookkeeping. Every firing (fired or blocked) is recorded as a passive `hook` event (`Recorder.record_hook`, template `record_refusal`) ‚Äî the hook mechanism itself stays recorder-free for the veto decision, matching `policy.py`. A hook that raises is caught, logged, and treated as `allow` (fail-open default; a security hook wanting fail-closed owns that in its own callback). See the README's Hooks section for the config format.

**Concurrent dispatch (issue #65).** The tool calls in one assistant message are parsed, then partitioned by `_plan_groups` into ordered execution groups: a maximal run of *contiguous* provably-read-only calls becomes one group the executor **overlaps** (bounded `ThreadPoolExecutor`), while every other call ‚Äî a parse error, or one whose effects can't be proven read-only ‚Äî is its own singleton group run inline in sequence. A non-overlappable call is therefore a **barrier**: nothing after it starts until it finishes, and it starts only once everything before it finished, so the observable ordering of the sequential path is preserved for every non-overlapped call, and read-only calls only ever overlap other read-only calls (never a write). Independence is decided by `concurrency.is_parallel_safe`, deliberately conservative ‚Äî a false negative only costs a missed speed-up, a false positive could corrupt shared state, so every rule errs toward sequential: `read_file`/`fetch_web` qualify for any args; `bash` qualifies only when `is_read_only_command` proves its `command` is a `|`-pipeline of allow-listed read-only commands with **no** redirection/chaining/substitution metachar (`> < ; & $( \``); `write_file`, `read_tool_artifact` (makes its own LLM call), `delegate` (a subagent can perform any CRUD mid-run ‚Äî deferred per #65's recorded limitation), and unknown tools never qualify. The worker pool is bounded by `MCA_TOOL_MAX_CONCURRENCY` (default 4; `1` disables overlap). Only the tool *invocation* (I/O-bound work) runs on worker threads (each under a `contextvars.copy_context()` so `current_session_id`/recorder propagate); argument prep, output offloading/dedup, envelope building, result assembly, and every recorder emit stay on the main thread **in call order**, so the shared tool-artifacts store, session files, and recorder capture state are never touched concurrently and each result still maps to its own `tool_call_id` regardless of finish order. Because the recorder's `before_tool`/`after_tool` share a single `_pending` slot (a strict before‚Üíafter pairing that overlap would race), the concurrent path skips `before_tool` and hands `after_tool` an explicit `(start_mono, end_mono, started_at)` timing bracket each worker captured, so per-call latency/resources reflect the isolated call, not the group's wall-clock.

`tool_execution.schema` is the single, centrally-configured home for every offload-related size threshold (`ARTIFACT_THRESHOLD`, `MAX_TOOL_OUTPUT_CHARS`, `PAGE_FETCH_MAX_CHARS`, `PREVIEW_TOKEN_BUDGET`/`PREVIEW_MAX_CHARS`) ‚Äî `tool_registry` reads them from there rather than each package defining its own copy. Because `tool_execution/__init__.py` also imports `ToolRegistry`/`artifact_file_path` from `tool_registry` (to dispatch calls and locate artifact files), a naive two-way top-level import would cycle; `ToolExecutor` and `args.strip_unknown_args` instead import `ToolRegistry` lazily (inside the function/method that needs it, or via `type(self.registry)` once an instance exists) so `tool_registry` can safely import the shared constants from `tool_execution.schema` at module load time.

When a tool output is large enough to offload ‚Äî `bash` combined stdout+stderr, a `read_file` file, or a `fetch_web` page above `ARTIFACT_THRESHOLD` ‚Äî the tool returns the `(None, artifact_dict)` contract (file/page content in the `stdout` slot) and `after_tool_call` bounds **each stream independently** (`_offload_streams`): a stream over the preview budget (`PREVIEW_MAX_CHARS`) has its **full content written to its own per-stream file** at `.my_coding_agent/<session>/artifacts/<tool_call_id>.<stream>.txt` (`stream` ‚àà `stdout`/`stderr`) at creation time and its envelope field carries only a **bounded preview** ‚Äî a token-bounded excerpt + inline guidance; a stream within budget is inlined. The guidance is **structure-aware** (`lang.looks_like_json`, a deterministic content sniff ‚Äî no LLM call): when the offloaded text is JSON and an on-disk file exists, the guidance leads with a mechanical path (`jq`/`python` over that file) as the primary move, ahead of `read_tool_artifact(tool_call_id, query=...)` and byte-range retrieval ‚Äî including for single-line JSON, which would otherwise fall into the byte-range-only warning; non-JSON content keeps the prior guidance (bash text tools as a secondary path, byte-range-only for single-line). Every offload's guidance also discloses that `read_tool_artifact(query=...)` is a separate, output-token-capped LLM call that may scan only part of a large artifact (the cap number is read from `EXTRACTION_OUTPUT_TOKEN_BUDGET`, canonical home `tool_execution/schema.py`), so choosing it is an informed trade rather than a blind default. `output` carries the stdout preview/inline and `error` the stderr preview/inline, so a large stderr can no longer flood context. `metadata.preview` is keyed by offloaded stream (`{"stdout": {‚Ä¶}, "stderr": {‚Ä¶}}`, shown/total counts + `full_output_path`). The full raw stream never enters the context window; because each file persists for the run, any later step can query it. The path scheme (and its safe-id guard) lives in one place, `tool_registry.artifact_file_path(session_id, tool_call_id, stream)`, shared by the write side (executor) and the read side (`read_tool_artifact`) so the two can never drift. The on-disk write is fault-tolerant: if `mkdir`/`write_text` fails (full disk, permissions), `_write_artifact_file` logs a warning and returns `None` instead of aborting the run ‚Äî offloading and the preview continue without an on-disk copy, and retrieval falls back to the in-memory store. The preview excerpt is built by `output.build_stream_preview`, applied per stream.

Before writing that per-stream file, `_offload_stream` checks the offload-bound text against this run's already-stored artifacts (`_find_duplicate`, scanning `.my_coding_agent/<session>/artifacts/*.txt` newest-first ‚Äî deterministic, no LLM): an exact `sha256` hash match, or failing that, containment (the text is a contiguous substring of a stored artifact of equal-or-larger size ‚Äî needed because `bash` rstrips its own streams, so a read-back of a file with trailing whitespace differs from the original by exactly that). A hit writes **no new file and no preview**; the envelope field instead carries a short pointer naming the original `tool_call_id`/stream/offset/length, and `metadata.duplicate_of` carries the same as a structured, per-stream descriptor (`{"stdout": {tool_call_id, stream, offset, length}, ...}`) ‚Äî absent when nothing deduplicated. Streams are independent: a duplicate stdout with a novel stderr dedups the former and offloads the latter normally. This closes the read-back loop where the model re-reads an artifact file (e.g. via `bash cat`) and the bytes get offloaded again as a new artifact (issue #92).

**Bash OS-level sandbox (`engine/sandbox/` package, issue #25).** Opt-in confinement for `bash` subprocesses on macOS, enforced by the OS beneath the shell rather than by inspecting the command text ‚Äî the same category of guarantee the refusal gate above cannot give (obfuscation defeats a textual gate by design; a kernel-level boundary does not). `sandbox.is_enabled()` reads `MCA_BASH_SANDBOX` (opt-in, off by default), the same env-var propagation pattern as the refusal gate's `MCA_DISABLE_DANGEROUS_COMMAND_GATE` ‚Äî set once by the CLI's `--sandbox` flag before the run starts, read at call time, so a `delegate`-spawned subagent (same process) inherits the exact posture for free with no `RunContext` threading. `registry.bash` checks `sandbox.is_enabled()` first: off, the path is untouched (`shell=True` exactly as before, byte-identical). On, `sandbox.probe_host_capability()` gates fail-closed ‚Äî non-macOS or a missing `sandbox-exec` binary returns an `ok:false` envelope naming the reason and runs no subprocess at all, never falling through to unconfined execution. Otherwise `sandbox.default_scope(self._base_dir)` builds a `SandboxScope` (workspace root + a narrow temp allowlist: `/tmp`, `/private/tmp`, `$TMPDIR` ‚Äî deliberately not the whole `/private/var/folders` parent, which is shared machine-wide across every process's own temp subdirectory and would defeat the scope), `sandbox.wrap_command` generates a Seatbelt profile (`(deny default)`, broad `file-read*`, `file-write*` scoped to each write root via `subpath`, `(deny network*)`) and returns the `["sandbox-exec", "-p", <profile>, "/bin/sh", "-c", command]` argv, run with `shell=False` so Python doesn't double-interpret it ‚Äî `/bin/sh -c` still gives the command pipes/redirections/builtins exactly as `shell=True` did. Seatbelt confinement applies to the whole process tree by construction (a spawned child inherits it), so no extra plumbing is needed for that guarantee. A failed sandboxed command whose stderr contains `Operation not permitted` (the EPERM message a kernel-denied syscall surfaces, distinct from the EACCES `Permission denied` an ordinary Unix permission failure produces) is classified ‚Äî best-effort, observability-only, never enforcement ‚Äî as a likely sandbox denial (`sandbox.is_likely_denial`) and recorded via `Recorder.record_sandbox_denial(command, exit_code, stderr, step)`; an ordinary command failure (e.g. `grep` finding nothing) is not misreported as a denial. `AgentNode.execute` emits `Recorder.record_sandbox_activation(workspace_root, extra_write_paths)` once per run, immediately after `recorder.start`, only when the sandbox is enabled ‚Äî a sandbox-off run's `events.jsonl` stays byte-identical to a pre-change trace. Non-goals (deferred): the egress allow/deny policy itself (only the network default-deny boundary is established here), read-scope tightening for secret files, and container/VM sandboxing for fully-untrusted repos.

### `pipeline/` ‚Äî DAG Building and Execution

The node-based DAG execution engine. `pipeline/` only knows how to build and execute a DAG ‚Äî it has no knowledge of LLM client internals or session management.

**`RunContext` (`context.py`)** ‚Äî the explicit data contract that flows through the pipeline. Holds immutable run config (session id, max steps, LLM client, recorder, all tools) and mutable state fields (messages, step_num, last_prompt_tokens, tool_records, tool_artifacts, handoff_records, plus the skill snapshot and the run's `loaded_skills` set ‚Äî issue #19). Control signals (`signal`, `stop_reason`) are written by nodes and read by `Pipeline.execute`. `LLMCallNode`/`ToolDispatchNode` both consult `ctx.all_tools` directly (per #114) ‚Äî no per-step routed subset exists.

**`Node` protocol + `BaseNode` (`node.py`)** ‚Äî a `Node` is any callable with a `name: str` and a `run(ctx: RunContext) -> None` method. Nodes read and write `ctx` in place.

**`Pipeline` (`dag.py`)** ‚Äî takes an ordered list of `Node` objects. `run_step` executes every node in order for one step, short-circuiting when any node sets a non-`CONTINUE` signal. `execute` wraps `run_step` in the outer step loop.

**The five default nodes** (instantiated by `build_default_pipeline()`; tool routing was removed from the pipeline by issue #114 and the routing code itself deleted by issue #204 ‚Äî see above):

| Node | Stage | What it does |
|---|---|---|
| `ContextGuardNode` | 1 | Retires provably-superseded tool results (inlined in `pipeline/nodes/context_guard.py`, issue #121), then checks `last_prompt_tokens / context_window`; sets STOP (limit), RESET (handoff), or CONTINUE |
| `LLMCallNode` | 2 | Increments `step_num`, calls `chat_completion(ctx.messages, tools=ctx.all_tools)` (the full registered toolset, per #114), appends assistant message |
| `ToolDispatchNode` | 3 | Builds `ToolExecutor(last_message, ctx.llm)`, runs it, merges records and artifacts into `ctx` |
| `AnomalyDetectNode` | 4 | Scans `ctx.tool_records` for a same-signature failure streak (helpers inlined in `pipeline/nodes/anomaly_detect.py`); at the 3rd consecutive same-class failure (and each further one), records an `anomaly` event and logs one WARNING at first signal. Reads `ctx.tool_records`/`ctx.last_response` only; never touches `ctx.messages`/`ctx.signal`/`ctx.stop_reason`, makes no LLM calls ‚Äî detection-only, covering subagents for free since every delegated run builds its own pipeline instance |
| `FinalizeStepNode` | 5 | Reads `ctx.last_response`: records token usage + updates `ctx.last_prompt_tokens`, then sets STOP on a stop/exit/quit finish_reason (emits both `token_tracking` and `finish_check` records) |

**Tool-result supersession (issue #121).** Before its budget check, `ContextGuardNode` runs a deterministic, no-LLM pass (`find_retirements`, inlined in `pipeline/nodes/context_guard.py`) that retires tool results provably made obsolete by a later one, over `ctx.tool_records`/`ctx.messages`: Case A ‚Äî a `read_tool_artifact` extract carrying the incompleteness marker for artifact X, superseded once a later successful call reads the same X; Case B ‚Äî an earlier result's full text is a contiguous substring of a later result's text (window-side mirror of the #92 dedup rule); Case C ‚Äî an older invocation of a byte-identical `(name, args)` call, superseded by its newest successful invocation. Retirement replaces the tool message's content with a one-line stub (same role/`tool_call_id`, naming the tool, the superseding call, and the on-disk artifact path when one exists) as a **new message object**, never an in-place mutation or a delete ‚Äî so the recorder's identity check for that step's prefix fails and falls back to a full snapshot (fidelity over size, same invariant `_encode_messages` already guarantees), and checkpoint/resume and the sum-check are unaffected since neither inspects message content. A size floor (`SUPERSESSION_SIZE_FLOOR_CHARS`, `engine/tool_execution/schema.py`) skips trivial retirements that wouldn't outweigh the KV-cache churn a replacement causes, and `MCA_SUPERSESSION=0` disables the pass entirely, restoring today's append-only behavior byte-for-byte. Each retirement records one passive `supersession` event (`Recorder.record_supersession`) ‚Äî auditable per session, ignored by readers that predate the event type.

### `AgentNode` (`pipeline/nodes/agent.py`)

The top-level entry point. Holds an `LLM` client via composition (`self.llm`) ‚Äî not a subclass. `__init__` builds the client, assigns a session id, attaches the session log, and initializes run stats.

- **`execute(max_steps)`** ‚Äî stand-alone entry: constructs a `RunContext`, builds the pipeline via `build_default_pipeline(spawn_fn=...)`, delegates to `pipeline.execute(ctx)`, saves session data, and prints the summary. `max_steps` defaults to the shared `DEFAULT_MAX_STEPS` (50) ‚Äî the single source of truth used by the CLI, the `execute` default, and delegated subagents so all three share one step ceiling.
- **`run(ctx)`** ‚Äî embedded entry: runs `execute()` and writes results back to the provided `RunContext`; used when `AgentNode` is a step in a larger outer pipeline.

`AgentNode` owns session bookkeeping: banner printing, session log attachment/detachment, `session_data.json` + `tool_artifacts.json` persistence, run summary, and observability recorder start/finish. It also owns `_generate_handoff`, `_spawn_continuation`, and `_handle_context_reset` ‚Äî the context-reset machinery called by `ContextGuardNode` via the `spawn_fn` closure. A shared `summarize_conversation(llm, messages, prompt, kind)` module-level helper (`pipeline/nodes/context_summarizer.py`) backs both `_generate_handoff` and `generate_report` ‚Äî a single tool-free LLM call over the whole conversation; when the model returns empty `content` (reasoning models such as Qwen3-thinking often end the summary turn with a tool call, leaving `content` null and the substance in `reasoning_content`), it falls back to `reasoning_content` so the summary is never lost.

A delegated subagent's end-of-turn report is produced by exactly one of three cost-divergent paths, chosen in `delegate()` (`engine/tool_registry/registry.py`), in this order: (1) **verbatim** ‚Äî a clean finish with a non-empty final assistant turn hands that turn back as-is, zero extra LLM calls; (2) **summarizer** ‚Äî a step-ceiling cutoff hands back `ctx.handback_report`, already synthesized in-pipeline by `ContextSummarizerNode` as one `report`-kind LLM call; (3) **fallback** ‚Äî `generate_report()` is the out-of-pipeline last resort (aborted runs, or a clean finish whose final turn carries no usable text), issuing its own `report`-kind LLM call; because `execute()` already saved `session_data.json` before this call runs, `delegate()` re-saves the child afterward so its persisted `total_usage` includes the report's tokens. Every path records its identity as `source` (`verbatim` / `summarizer` / `fallback`, from `engine/schema.py`) on the `report` event via `Recorder.record_report(content, source)` ‚Äî a required parameter, so a report can never be recorded without declaring which path produced it ‚Äî making each report's cost statable from `events.jsonl` alone (report-cost-provenance).

**Run resilience & resume (run-resilience D3‚ÄìD6).** `AgentNode` owns a per-step resume checkpoint, entirely separate from `events.jsonl` (observability stays passive ‚Äî ¬ß25; execution never reads the event stream back). After each *completed* step the pipeline calls the engine's `_write_checkpoint(ctx)` (wired in as `build_default_pipeline(..., checkpoint_fn=...)`), which atomically writes `checkpoint.json` (`engine/checkpoint.py`: exact `messages` + `step_num` + `last_prompt_tokens`, via write-temp + `os.replace`). A step that raises never reaches the checkpoint call, so **a partial step is discarded** ‚Äî a run killed mid-step resumes from the end of the last completed step, and any tool side effects from the discarded partial step are simply unknown to the resumed conversation (same risk class as re-running a command after a crash; there is no step-level journaling). An unrecoverable LLM failure (tolerance exceeded, or a non-retryable class) is caught in `execute` as a first-class stop: it sets a classified `stop_reason` (`llm_failure_<classification>`), stashes the error on `self.failure_error`, and lets the existing `finally` persist `session_data.json`; the CLI (`cli.py`) then exits non-zero with a one-line resume hint and no traceback. Resume is `AgentNode.from_checkpoint(checkpoint, ...)`: it builds a **new** session (fresh id, clean trace) seeded with the checkpointed conversation and step counter so its first LLM call is step N+1, and records a `resumed_from` link (mirroring the delegation `parent_session_id` link) on the `session_start` event and in `session_data.json`. The dead session's files stay byte-identical ‚Äî resume never appends to the original.

**Known limitations (by design).** The end-of-run checkpoint cleanup keeps the "checkpoint exists iff genuinely resumable" invariant only for the *immediate* source, so two edges are accepted rather than coded around: (1) *Multi-hop resume chains* ‚Äî cleanup clears only the immediate `resumed_from` and only on a clean finish, and a `max_steps` hop deliberately keeps its checkpoint (still resumable with a larger budget); a completed 3+-hop chain with a `max_steps` middle hop (`W` dead ‚Üí `X` resumes `W`, hits `max_steps` ‚Üí `Z` resumes `X`, finishes) can leave an ancestor's checkpoint (`W`) behind, which `--resume-last` may then target to re-run an already-finished task (no progress lost; workaround: `--resume <id>` explicitly, or delete the stale `.my_coding_agent/<id>/checkpoint.json`). (2) *Propagated continuation-failure trace artifacts* ‚Äî when a post-context-reset continuation fails unrecoverably and the failure is propagated to the main run, the main session's `session_data.json` (`stop_reason=context_reset`, written at reset time) and its `session_end` event (`stop_reason=llm_failure_*`) can disagree ‚Äî a cosmetic observability inconsistency; the authoritative failure and resumable checkpoint live in the continuation session, which the CLI resume hint names.

`AgentNode` also accumulates a **rollup**: `child_rollups` collects each delegated subagent's usage summary (own per-kind totals, elapsed seconds, step count, its own nested descendants, and its report's `source`) as `delegate()` hands it up via the `current_agent_node` contextvar ‚Äî write-side, no re-reading of child files. `_usage_summary()` combines this agent's own per-kind totals with `child_rollups` into a `grand_total` (own + every descendant, transitively), persisted as `session_data.json`'s `rollup` key and surfaced in the terminal run summary as a task-level cost section (own vs. rolled-up totals, one line per direct subagent marked free/paid by its report `source`) whenever delegations occurred ‚Äî unchanged output otherwise. `viewer/sumcheck.py` provides a deterministic, LLM-free check over a session directory tree: per-call usage rows sum to the persisted `by_kind`, own totals plus descendants' grand totals equal `grand_total`, and a report's `source` is consistent with the presence/absence of a `report`-kind usage row ‚Äî exposed via `my-coding-agent-webui --check <session_id>` (exit 0/1; incomplete records report as unverifiable, not failed).

### Context Handoff (`pipeline/schema.py` + `pipeline/handoff.py` + `pipeline/nodes/context_guard.py`)

`pipeline/schema.py` holds the `ContextHandoff` dataclass shape only; `pipeline/handoff.py` holds the builder logic that operates on it ‚Äî `save_handoff()` (persistence) and `handoff_to_user_message()` (rendering) ‚Äî kept out of `schema.py` per the Schema Convention.

When `ContextGuardNode` detects that `prompt_tokens / context_window >= context_reset_threshold`, it calls `spawn_fn()` ‚Äî a closure set by `AgentNode.execute` that calls `AgentNode._handle_context_reset`. That method:
1. Sends a structured handoff prompt asking the LLM to summarize progress, files changed, decisions made, and remaining work.
2. Saves the result as a markdown file under `.my_coding_agent/handoffs/` via `save_handoff()`.
3. Saves session data and prints a summary for the current run.
4. Spawns a fresh `AgentNode` instance carrying only the system messages and the handoff as a user message.

`ContextGuardNode` stores the result in `ctx.continuation_messages` and sets `ctx.signal = "RESET"`, which causes `Pipeline.execute` to return those messages immediately.

### `ToolRegistry` (`engine/tool_registry/` package)

A plain class whose methods are the tools the LLM can call:

| Tool | Purpose |
|---|---|
| `bash(command, timeout, stdin)` | Runs a shell command; reports `stdout, stderr, exit_code, ok`; large combined output offloads. `stdin` delivers text to the command's standard input directly (subprocess `input=`), bypassing shell composition ‚Äî the structural channel for multi-line scripts (written via `write_file`, run with `bash`, data via `stdin`) instead of pipe+heredoc or quote-nested `-c` one-liners. When the OS-level sandbox is enabled (see below), the command runs as `sandbox-exec -p <profile> /bin/sh -c <command>` (`shell=False`) instead of `shell=True` directly |
| `read_file(file_path)` | Reads a file; files over `ARTIFACT_THRESHOLD` offload (lossless ‚Äî the full content is stored, not truncated) |
| `write_file(file_path, content)` | Writes a file, creating parent dirs |
| `fetch_web(url)` | Fetches any text URL; converts to markdown only for HTML responses (`text/html`, `application/xhtml+xml`) ‚Äî every other text type (JSON, plain text, XML, ‚Ä¶) passes through verbatim, and non-text types (`image/*`, `application/octet-stream`, ‚Ä¶) are rejected with an explicit error. | `read_tool_artifact(tool_call_id, query, start, length)` | Two mutually exclusive modes. Query mode (default, `query` required): makes a bounded local-LLM extraction call (`artifact_query` kind) over the stored content and returns only a token-bounded extract relevant to `query` ‚Äî never the full stored content, regardless of query, artifact size, or failure mode. Artifacts over the per-call input budget are scanned in sequential chunks so any part of the output is reachable, not just a prefix. Degrades to a bounded head excerpt + on-disk path hint when no LLM client is injected or the extraction call fails. Any incompleteness in query mode ‚Äî a chunk completion cut at the extraction token cap, the joined result sliced to the output budget, or chunks left unscanned when the budget fills ‚Äî is disclosed as a trailing bracketed marker in the returned text naming the cause, the scanned-vs-total extent, and a recovery path (narrower re-query, byte-range retrieval, or bash-skim of the on-disk file); a complete extract carries no marker. Cap detection tolerates a provider that omits `finish_reason` on a capped completion (`finish_reason == "length"`, or an empty reason with `completion_tokens` at the budget). The completion cap in force for an extraction call is also recorded on its `llm_call` trace event so the viewer can badge a cut `artifact_query` call without importing the engine's budget constant. Range mode (`start` given, `query` ignored): returns an exact, verbatim byte-range slice ‚Äî `start`-based, `length` defaulting to and capped at `RANGE_MAX_CHARS` (aliased to `PREVIEW_MAX_CHARS`) ‚Äî prefixed by a one-line range/total header, no LLM call; the only mode that works on content with little or no line structure, and what a `duplicate_of` pointer's offset/length feed into. | `delegate(task, known_facts="", expected_report="")` | Spawns a fresh read-only subagent for a focused task; the subagent inherits the parent toolset **minus `delegate`** (to prevent recursive spawning) and runs with the same `DEFAULT_MAX_STEPS` budget as the main agent. `known_facts` is optional and additive-only ‚Äî facts the task text doesn't already state (file paths, environment facts, constraints); its schema description tells the parent never to restate the task there, and a deterministic per-line lexical-containment guard (`strip_task_restatements`) drops any line that duplicates or near-duplicates the task before the child's opening message is built, so a restatement that slips past the schema guidance never reaches the child's prefix. `expected_report` is optional and describes the *output* instead ‚Äî the shape, size, and content the parent needs back, so it can drop the child's final message directly into its own answer instead of re-synthesizing it; the child's system prompt states that its final turn is handed back verbatim and must follow any stated expectations. The child's opening message is task-only when both are absent, or `"Task:\n{task}"` with a `"Known facts from the main agent:\n{facts}"` section and/or a `"Report expectations from the main agent:\n{expected_report}"` section appended, each only when non-empty. Returns the subagent's report by cost order: a clean finish's final assistant turn verbatim (zero extra LLM calls); otherwise the in-pipeline synthesized `handback_report` (one `report`-kind call already made); otherwise `generate_report()`'s out-of-pipeline fallback call ‚Äî never a stale scrape of an earlier message, so the final tool results survive even when the subagent is cut off at its step ceiling. `use_skill(name)` is the one conditionally-registered tool (issue #19): it is on `ToolRegistry` but excluded from the automatic public-method scan, and only added to the toolset when ‚â•1 skill is discovered ‚Äî so a skill-free run's tool schemas are byte-identical to before. It returns the named skill's body (prefixed `Skill: <name>`) through the standard envelope path (oversized bodies offload via the same artifact machinery as `read_file`); an unknown name returns an `Error:`-prefixed string (‚Üí `ok:false` envelope) listing available names; a name already in the run's loaded-set returns a short "already loaded" pointer instead of re-injecting the body. It is deliberately **not** in `concurrency.is_parallel_safe`'s allow-list because it mutates the loaded-set.

The `@tool` decorator converts any `ToolRegistry` method into an OpenAI-compatible tool definition by inspecting its signature and parsing Google-style docstrings.

### Skill Knowledge Delivery (`engine/tool_registry/skills.py`, issue #19)

User-authored procedural knowledge reaches the model without editing Python source. A *skill* is a directory with a `SKILL.md` (minimal `name`/`description` frontmatter split by a ~20-line hand-rolled parser ‚Äî no YAML dependency ‚Äî plus a free-Markdown body). `discover_skills()` scans `<cwd>/.my_coding_agent/skills/*/SKILL.md` (project) then `~/.my_coding_agent/skills/*/SKILL.md` (user) **once** at session start, project shadowing a same-named user skill, returning a name-sorted snapshot; a malformed skill is skipped with one warning, never a failed run. `render_skill_index()` turns the snapshot into a compact index (fixed header naming `use_skill` and stating the precedence contract ‚Äî skills guide tool usage but never override safety rules ‚Äî then one `- name: description` line per skill) bounded by the fixed char caps in `tool_execution/schema.py` with two deterministic degradation tiers (even description truncation ‚Üí names-only); it never exceeds the total cap and uses no LLM or ranking.

**Placement is on the opening user message, never the system prompt** (so the #75 prefix-cache invariant holds trivially): `AgentNode` appends the index block after the task text of the first user message at construction (`_place_skill_index`), and emits the `skill_index` *offered* event at session start. When the snapshot is empty it appends nothing ‚Äî the opening message and tool schemas stay byte-identical to before. The discovered-snapshot and the run's `loaded_skills` set ride on `RunContext` (the set shared by reference into each per-message `ToolRegistry` so `use_skill` dedup persists across steps). On a context reset the continuation carries the same snapshot and a copy of the loaded-set, re-injecting each loaded skill's full body after the index so knowledge survives the handoff; `delegate` passes the parent's snapshot to the child (no disk re-scan) but the child keeps its own empty loaded-set. `use_skill` is registered conditionally ‚Äî only when skills are discovered (`cli.py`'s `_build_tools`) ‚Äî so it is absent when no skills are registered.

### `observability/` ‚Äî Passive Event Capture

Receives events emitted by `engine/` and `pipeline/`; never controls execution. Writing directly to `events.jsonl` is its only side-effect.

- **`sampler.py`** ‚Äî `ResourceSampler`, a background daemon thread sampling machine-wide RAM/CPU/network/disk (`psutil`, no privileges) and Apple Silicon GPU utilization (`ioreg -r -d 1 -c IOAccelerator`, parsing `PerformanceStatistics` ‚Üí `Device Utilization %` ‚Äî the only sudo-free GPU signal on macOS; `powermetrics` needs root) into a bounded in-memory ring buffer at a ~1 s interval. `summarize_window(start, end)` returns gauges (RAM/CPU/GPU) as avg+peak and counters (network/disk) as edge deltas over a monotonic window, falling back to one on-demand snapshot when the window is shorter than the sampling interval; `session_rollup()` returns the same shapes over the whole run, folding samples evicted from the ring into a running total so long runs stay correct. Any sampling or snapshot exception disables further capture and logs one warning ‚Äî never raises into the run. `get_sampler()` returns one process-wide singleton; `start`/`stop` are reference-counted so the main agent and in-process subagents (sharing identical machine-wide data) don't tear the thread down under each other.
- **`recorder.py`** ‚Äî event type constants (`SESSION_START`, `LLM_CALL`, `TOKEN_TRACKING`, `FINISH_CHECK`, `HANDOFF`, `REPORT`, `ANOMALY`, `REFUSAL`, `EGRESS`, `SANDBOX_ACTIVATION`, `SANDBOX_DENIAL`, `EXFIL`, `SKILL_INDEX`, plus the run-resilience `LLM_WAIT`/`LLM_RECOVERY`/`LLM_FAILURE`, etc.); `Recorder` appends events as newline-delimited JSON. `record_refusal(tool_name, command, rule_id, reason, references, safer_alternative, step)` follows `record_anomaly`'s template ‚Äî purely passive, reporting what the `tool_execution.policy` gate already decided (via the executor) and never judging or influencing execution; absent entirely from a refusal-free run (dangerous-command-refusal-gate, issue #124). `record_egress(tool_name, host, matched_list, reason, step)` follows the identical template for the network-egress-filter (issue #126): purely passive, reporting what `engine.egress` already decided (via the executor), absent entirely from a block-free run. `record_sandbox_activation(workspace_root, extra_write_paths)` and `record_sandbox_denial(command, exit_code, stderr, step)` follow the same passive template (bash-os-sandbox, issue #25) ‚Äî absent entirely from a sandbox-off run. `record_exfil(tool_name, category, step)` follows the same template for the secret-exfiltration guard (issue #127) ‚Äî purely passive, and never receives (so never can leak) the matched secret value, only its category name; absent entirely from a block-free run. The `session_start` event carries an optional `resumed_from` (the dead session a `--resume` run continues, run-resilience D5), mirroring `parent_session_id`, plus `posture` (issue #130) ‚Äî `"sandboxed"` when the #25 OS sandbox is active, else `"screened_only"`, derived once by `policy.get_protection_posture()` and passed straight to `Recorder.start`; pre-change traces simply lack the key; `record_llm_wait`/`record_llm_recovery`/`record_llm_failure` are additive, passive rows the LLM client emits during outage recovery ‚Äî the reader ignores unknown event types, so they never disturb an existing trace load or the sum-check. `record_skill_index(names, chars, tier)` records that a skill index was placed into an opening user message ‚Äî once at session start and once per continuation (the *offered* record, issue #19); it is absent entirely from a skill-free run's stream, so pre-skill traces parse identically. `use_skill` invocations are captured as ordinary `tool_call` events; loaded / deduplicated / error outcomes are distinguishable from the envelope alone (full body vs "already loaded" pointer vs `ok:false`). `Recorder.start`/`finish` start/stop the shared `ResourceSampler`; `resource_window(start, end)` and `resource_rollup()` are thin pass-throughs so call sites don't import the sampler directly. `record_llm_call` and `record_summarizer` accept an optional `resources` object (the caller's sampler-window summary for the call's bracket) written onto the event only when not `None`; `after_tool` computes its own window from the pending tool-start time it already tracks internally ‚Äî except on the concurrent dispatch path (issue #65), where the executor passes an explicit `(start_mono, end_mono, started_at)` bracket via `timing=` and the shared `_pending` slot is left untouched so overlapping calls can't race it. `record_anomaly(kind, streak_id, signature, tool_name, streak_len, tokens_spent, step)` is purely passive ‚Äî it records what `AnomalyDetectNode` reports and never judges or influences execution; rows sharing one `streak_id` let consumers keep only the latest (final) magnitude for a streak. Every timed event (`llm_call`, `tool_call`, `summarizer`) records a **true start timestamp**: the caller captures wall-clock time alongside its monotonic latency timer (`time.monotonic()`) and passes it through as `started_at`, rather than the recorder deriving it from `now()` at emit time ‚Äî so `started_at + latency_s` fits within the session's own bounds instead of the stamp mislabeling completion time as start. `now()` (the emit-time fallback, and the source for untimed events; public so `engine/tool_execution` can obtain recorder-consistent timestamps for its own concurrent-dispatch bracket without crossing the domain boundary through a private symbol, issue #180) formats ISO-8601 with millisecond precision and a UTC offset (`datetime.now().astimezone().isoformat(timespec="milliseconds")`); the reader tolerates the pre-change format (second precision, naive) since it never parses timestamps, only stores them. `record_llm_call` captures each call's full input ‚Äî the `messages` snapshot and the `tools` definitions given to the model that turn ‚Äî for kinds in `FULL_PAYLOAD_KINDS` (both dropped otherwise to bound stream size). To keep `events.jsonl` growing linearly instead of quadratically with conversation length, the `messages` snapshot itself is delta-encoded per call kind (`_encode_messages`, incremental-trace-capture D1/D2): the recorder keeps, per kind, the last emitted snapshot's physical call number and message-object references; on the next call of that kind, if the current messages provably extend that snapshot's prefix (verified by `id()` identity over the shared region ‚Äî never by content comparison, and never mutated in place per the append-or-replace invariant on `ctx.messages`), the event carries `messages: null` plus `messages_base_call`/`messages_prefix_len`/`messages_suffix` instead of the full list. Any identity mismatch, a shorter list, or the first call of a kind falls back to a full snapshot ‚Äî fidelity always wins over size. `record_report(content, source)` captures a subagent's end-of-turn final report as a `REPORT` event, distinct from the context-reset `HANDOFF` event; `source` (required, from `engine/schema.py`'s `REPORT_SOURCE_*` constants) names which of the three cost-divergent paths produced it. `after_tool(name, args, result, ok, error, timing=None)` requires the outcome the executor already holds (never re-derived by parsing `result`): every `tool_call` event carries a top-level `ok`, and on failure (`ok: false`) an `error` text plus an `error_class` computed by the shared helper `observability.classify_error` (the same normalization `pipeline/anomaly.error_signature` delegates to, so the two classifications agree by construction) ‚Äî a consumer answers "which calls failed, with what class" with one `json.loads` per line, no parsing of `result`. Pre-change traces simply lack these fields; the reader falls back to parsing `result` only for those. Three `ContextVar`s (`current_session_id`, `current_recorder`, `current_agent_node`) let delegated subagents record their parent link and hand usage totals up to the parent `AgentNode` without re-reading child files.

### `viewer/` ‚Äî Active Read-Side (Trace Explorer)

The read-side of the observability system. Separated from `observability/` because it is **active** ‚Äî it controls execution (HTTP server), renders output (embedded HTML), and manages file handles ‚Äî whereas `observability/` is passive capture only (CONTRIBUTE.md ¬ß25). `viewer/` serves the Trace Explorer directly: `server.py` owns the HTTP server and the `my-coding-agent-webui` CLI (`--port`/`--dir`/`--check`) with no separate shell package in front of it (issue #206 dissolved the `webui/` shell that used to mount this as its sole tab, after #194 removed the Evals/Admin tabs).

- **`schema.py`** ‚Äî `TraceNode` and `TraceSession` dataclasses: the typed contracts produced by `reader.py` and consumed by `server.py`. `TraceSession.posture` (issue #130) carries the run's protection posture straight off `session_start`, or `None` for a pre-change trace. `TraceSession.verdict` (issue #198) carries the session's eval verdict ‚Äî `run_id`, `case_id`, `passed`, `metrics`, `detail`, `result_path` ‚Äî read from a `verdict.json` sibling when present, or `None` for a non-eval or pre-eval session; the field is an opaque, lightly-validated document owned by `evals/` (issue #196), not a mirrored dataclass. Machine-wide `resources` (when present) rides in `TraceNode.attributes["resources"]` for `llm_call`/`tool_call`/`summarizer` nodes ‚Äî no dedicated field, since `attributes` is already the free-form scalar-metadata bag; the run-level rollup rides in `TraceSession.analytics["resource_rollup"]`. `TraceNode.ctx_state` holds the per-node context-window snapshot ‚Äî cumulative `composition` by `system`/`user`/`assistant`/`tool` role, the per-role `added` this node appended (with `added_total`/`removed` and an `estimated` flag), plus `tokens`/`window`/`pct`. Each node also carries `agent` (owning session id) and `depth` (call-tree nesting level), so sub-agent traces nest under the main agent. `TraceSession.order` is the execution-order node spine the UI walks for keyboard navigation.
- **`pricing.py`** ‚Äî model price table (USD per 1M tokens), `compute_cost()` helper for the run's own model, and `project_costs(prompt_tokens, completion_tokens)` which projects those same token counts onto every non-zero-priced (hosted) row in `PRICING`, so a local run's recorded tokens can be budgeted against Claude/GPT/etc. prices; missing token counts project to `$0` rather than erroring, and zero-priced/unknown models never appear in the result.
- **`reader.py`** ‚Äî `_read_events` resolves any prefix-delta-encoded `llm_call` events (`_resolve_message_deltas`) before anything else touches the parsed rows: it walks events in file order, keyed by physical call number, reconstructing `base[:prefix_len] + suffix` (chaining through a base that is itself a delta) so every downstream consumer sees the exact full `messages` snapshot the call saw ‚Äî byte-identical to full-snapshot capture, both for legacy traces (detected by `messages` already non-null) and new delta traces. A delta whose base call number is absent from the file (corrupt/hand-truncated mid-event) degrades to that event's `messages` being `None` rather than failing the load; any prefix of a crash-truncated file still reconstructs every event fully written before the truncation, since a delta only ever references an earlier line in the same file. Parses `events.jsonl` into a **flat** `TraceSession`: every pipeline `BaseNode` subclass (`ContextGuardNode`, `LLMCallNode`, `ToolDispatchNode`, `FinalizeStepNode`) becomes one `TraceNode` in a single chain off the session root ‚Äî there is no `step` wrapper node; the step number is carried as an attribute. `_group_into_steps` anchors each step bucket on its main `llm_call` event for new sessions, or on a legacy `router` event for archived sessions recorded before #114 (dual-anchor rule, so archived grouping stays byte-identical while new sessions still group correctly without any router events). A subagent's `REPORT` event becomes a distinct `report` node (`Subagent Report`), separate from the context-reset `handoff` node; its `source` attribute (verbatim/summarizer/fallback) surfaces as `"unknown"` when absent (pre-provenance traces), never an error or a guessed path. Reconstructs the context window as four role buckets in execution order (`_assign_ctx_state`): each node contributes the message(s) it appends ‚Äî the session seeds system + opening user, an LLM call adds its `assistant` output (exact `completion`), a tool dispatch adds its result (character-estimated via a session tokens/char ratio, since tool tokens are never recorded); composition re-anchors to each LLM call's real input snapshot (`_role_split` splits the provider's flat `prompt_tokens` across the four roles by character share). Each agent gets its **own** context window: `_assign_ctx_state` processes only its session's nodes, so delegate sub-agents (loaded recursively at an incremented `depth` and grafted inline) keep the independent windows computed by their own load. `_compute_analytics` sums token totals across **every** call kind (not just `main`) across the grafted parent+children graph, and adds `by_kind` (tokens per call kind) and `by_agent` (per session id: tokens, call count, and wall time read off that agent's own `session_end.elapsed_s`) ‚Äî `llm_call_count` still counts `main` calls only, for step semantics. It also adds `analytics["projected_costs"]` via `pricing.project_costs` on those same rolled-up totals ‚Äî a read-time-only projection of the run's actual (possibly local, $0) tokens onto hosted-model prices, introducing no extra traversal; each `llm_call` node's `attributes["projected_costs"]` carries the same projection for just that call's own tokens, so per-call figures sum to the session figure by construction. `_read_resource_rollup` reads `resource_rollup` straight out of `session_data.json` in the same session directory (written once the run finishes) and, when present, folds it into `analytics["resource_rollup"]`; missing file or missing key both resolve to `None` without error, so a session without resource capture (or still in progress) surfaces no rollup. `_read_verdict` (issue #198) side-reads `<session_dir>/verdict.json` the same way ‚Äî absent file resolves to `None`, and an unreadable/non-JSON/non-object file also resolves to `None` after logging one warning, so a corrupt sibling never fails the session load; the result rides straight onto `TraceSession.verdict`, read-only and unmodified. Timestamps are never parsed (`started_at`/`ended_at` stay opaque strings), so pre-change traces and crash-truncated traces (missing `session_end`, or missing `session_data.json` entirely ‚Äî falls back to a two-node stub) load without error, with analytics computed purely from the event rows present. Also does loop detection. `anomaly` rows become one `anomaly` trace node per `streak_id` (`_add_anomaly_nodes`, keeping only the latest/final-magnitude row per streak), and `_flag_anomalies` sets `anomaly_flag` on the streak's own failing `tool_call` nodes via a positional back-walk (matching tool name + a failed result ‚Äî read from the node's top-level `ok` attribute when the event recorded one, else parsed from the raw JSON `result` for pre-change traces; `tool_call` events carry no `tool_call_id` to link exactly), separate from and alongside the existing `loop_flag`. `_build_tool_node` also sets `refusal_flag` directly from that exact call's own recorded envelope (`_refusal_metadata` parses the `tool_call` event's `result` for a `metadata.refusal` key) ‚Äî an exact per-call fact, unlike anomaly's positional back-walk, so no matching heuristic is needed; absent on any pre-change trace or a call the policy gate never refused (dangerous-command-refusal-gate, issue #124). `_compute_analytics` adds a `refusal_count` alongside `loop_count`/`anomaly_count`. `load_session` also copies `session_start`'s `posture` straight onto both the session-root node's `attributes` and `TraceSession.posture` (issue #130) ‚Äî no derivation, just a pass-through; absent (`None`) on any pre-change trace.
- **`server.py`** ‚Äî minimal stdlib `http.server` with three routes (`/`, `/api/sessions`, `/api/sessions/{id}`) and an embedded single-page Trace Explorer UI built with **Preact + htm** (vendored offline under `_vendor/`, injected inline ‚Äî no CDN, no build step). The UI is a nested call-**Tree** (Main Agent at the root; each `delegate` spawns a collapsible **Subagent** group, with a coloured rail, nested where it was called ‚Äî derived from each node's `agent`/`depth`) with keyboard navigation (auto-select), a type filter, and a single per-node **Context window** box (a system/user/assistant/tool composition bar + legend and the running total/%, badged with the owning agent for sub-agents since each tracks its own window); Each tree row shows the node's compact badges then its contribution summary (e.g. *+196 assistant*) ‚Äî including grouped rows (any node whose dispatch nests children, e.g. a `delegate` tool_call nesting its subagent session root), which carry their own ctx-window contribution just like leaf rows and must render it too. Node detail leads with a **uniform header** shared by every node type ‚Äî a single big **main badge** naming the node class (e.g. `ToolDispatchNode`, the `(tool)` suffix stripped since it appears as its own badge) and a badge row ordered by importance left‚Üíright: identity (tool name / llm kind / model) ¬∑ status (‚úì/‚úó, colored) ¬∑ colored type signals (üì¶ artifact, ‚úÇÔ∏è truncated, ‚úÇÔ∏è cut at N-token cap for a capped `artifact_query` llm_call, üìú multi-line for a `bash` call with a non-empty `stdin` or a newline in `command`, üß≠ phase, üÜì/üí∞/‚ùî free/paid/unknown for a `report` node's `source`, üñ• machine-wide cpu/ram/gpu figures for the node's window) ¬∑ counts (N tools / N calls / N steps) ¬∑ then de-emphasised housekeeping (‚ö° latency, üïò timestamp, step); only badges whose data exists are shown, and the same `nodeBadges` set drives the compact tree badges. A flagged `tool_call` node's header also shows a distinct `anomaly` tag next to (never replacing) the existing `loop` tag, and the stats strip shows an anomaly count alongside the loop count when either is present, plus a `machine (run)` breakdown row surfacing the session's resource rollup when captured. A refused `tool_call` node carries its own distinct `refused` tag ‚Äî shown in both the tree row and the detail header, alongside (never replacing) `loop`/`anomaly` ‚Äî and the stats strip shows a `üõë refused` count when the session has any (dangerous-command-refusal-gate, issue #124); the stats strip also shows a distinct `üîí sandboxed`/`üõ° screened only` posture tag from `data.posture` (issue #130), and shows neither on a pre-change trace; its detail panel renders a dedicated block (reason, matched rule id, standard reference(s) as clickable links, safer alternative) so the full explanation reads without opening `raw_envelope`. and a `projected on:` breakdown row showing the session's `projected_costs` per hosted model (distinct from the actual `cost_usd` shown in the header, which stays $0 for local runs). Each `llm_call` node also carries its own `projected_costs` in its attributes, visible in the detail panel's raw Attributes JSON ‚Äî no dedicated per-call rendering block, keeping that panel uncluttered. A `use_skill` `tool_call` node carries a **üßÝ skill** badge, and the stats strip shows the session's skill **offered / loaded** counts (`skill_offered_count`/`skill_loaded_count` from `_compute_analytics`) when the run had any skill activity ‚Äî both absent for pre-skill traces, which render unchanged (issue #19). When `data.verdict` is present (issue #198), the stats strip also shows a single `filter-btn`-sized **Eval: pass**/**Eval: fail** toggle button (colored via `verdict-btn`, no separate tag); expanding it renders a `VerdictDetail` block (run id, case id, metrics chips, and the detail/rationale content through the same `ContentBox` used elsewhere) ‚Äî absent entirely, with no button, for a session with no verdict. Below the header it is type-aware: a tool dispatch renders four labelled boxes ‚Äî **command**, **output**, **error**, and the always-shown **raw_envelope** (`{schema_version, tool, ok, output, error, metadata}`) ‚Äî each highlighted in its `metadata.lang` language, and empty output / absent error omitted so the raw envelope still guarantees an empty `output` whose real signal lives in `error` is never hidden. LLM calls render response/reasoning/tool-calls boxes, and a `report` node renders the subagent's final report content. An LLM call's **Inputs** surface both the `messages` snapshot and the `tools` definitions the model was given that turn. Every content box renders through a **`ContentBox`** dispatcher (issue #112): a value that is JSON (object/array or JSON-looking string) or carries an explicit `python`/`shell`/`json` language hint renders as before through the read-only `CodeBox` component wrapping a **CodeMirror 6** editor; every other free-text string ‚Äî LLM response/reasoning, report content, tool command/output/error without a hint ‚Äî renders through **`MarkdownBox`** instead. `MarkdownBox` runs the content through vendored **markdown-it** (`html:false`, so embedded HTML in the source is escaped rather than emitted as live markup) then vendored **DOMPurify** before the sanitized HTML is assigned to `innerHTML` ‚Äî the only insertion path, and the sole XSS defense for untrusted LLM output ‚Äî memoized on content so unrelated re-renders don't re-parse. Fenced code blocks inside the rendered markdown are highlighted by mounting a small read-only CodeMirror view (reusing `CB_THEME`) when the fence language is one CM6 already knows (`json`/`python`/`shell`); other languages fall back to the plain `<pre><code>` DOMPurify passed through. Rendered is the default view; a per-box **Rendered ‚áÑ Raw** toggle switches to `CodeBox` showing the byte-exact source (copy/find/line-numbers unchanged). `CodeBox` itself takes an explicit language (`json`/`python`/`shell`/`text`, from the backend `metadata.lang` hint) or falls back to JSON-vs-text inference, and provides syntax highlighting, line numbers, folding, a JSON schema breadcrumb derived from the caret's syntax-tree path (clickable to jump; JSON only), collapse-all/expand-all, copy-all, and `@codemirror/search` find (open + Enter/Shift+Enter next/prev). Editors are created only for the selected node's visible boxes, so many-node sessions stay fast. This module also owns the `my-coding-agent-webui` console script (`main`, `--port`/`--dir`/`--check`): `--check <session_id>` runs the deterministic sum-check via `viewer/sumcheck.py` and exits 0/1 with no server started, otherwise `run_server` binds `_TraceHandler` and blocks until Ctrl-C. This handler serves only `/`, `/api/sessions`, and `/api/sessions/{id}` ‚Äî no separate shell process sits in front of it (issue #206).
- **`_vendor/`** ‚Äî third-party UI libraries (Preact, Preact Hooks, htm as UMD bundles; CodeMirror 6 as a prebuilt IIFE bundle exposing `window.CM6` with `json`/`python`/`shell` languages ‚Äî shell via `@codemirror/legacy-modes`, no `lang-shell` package exists; markdown-it as a prebuilt IIFE bundle exposing `window.markdownit`; DOMPurify as a prebuilt IIFE bundle exposing `window.DOMPurify`) vendored offline so the localhost viewer needs no internet. JS only; excluded from coverage.

### `evals/` ‚Äî Repeatable Case Runner and Result Store (issue #139)

A leaf package: depends one-way on `engine`/`pipeline` (to drive `AgentNode.execute`) and never the reverse. Turns "did this change help or hurt?" from anecdote into a repeatable, scored, persisted answer.

All exception classes here (`UnknownScorerError`, `RubricError`, `JudgeError`, `EvaluationError`, `DatasetVersionMismatchError`) inherit `utils.exceptions.MyCodingAgentError` per ¬ß29 (issue #179), so a caller catching the project-wide base handles evals failures too; a conformance test (`tests/test_exceptions.py`) enforces this for every exception class under `src/my_coding_agent/`.

- **`schema.py`** ‚Äî `EvalCase` (id, task prompt, scorer ref, expected/threshold payload, optional dataset ref + tags ‚Äî constructed internally by `run_config_file.py` per config, never loaded from a file) and `EvalScore` (case id, pass/fail, numeric metrics dict, per-check detail, optional `session_id` ‚Äî additive, `None` on records predating it, issue #193) frozen dataclasses; `RESULT_SCHEMA_VERSION`. Stdlib-only leaf, same convention as `engine/sandbox/schema.py`.
- **`scoring.py`** ‚Äî `Scorer` protocol (`score(case, run_result) -> EvalScore`); `RunResult` (final output text + a trace handle ‚Äî the run's session dir/id, not a parsed `TraceSession`, so the baseline scorer doesn't pay for `viewer.reader` parsing it doesn't need); `ExactMatchScorer`, the one baseline deterministic scorer (`equals`/`contains` check on the final output); a module-level registry (`resolve_scorer`/`register_scorer`) keyed by the case's scorer ref ‚Äî the single extension point future trajectory (#140) and judge (#103) scorers register against without changing the runner.
- **`runner.py`** ‚Äî `_build_tools()`/`_final_output()`: the minimal system-prompt-free toolset and final-assistant-message extraction shared by a config-driven run (`run_config_file.py`), kept separate from the CLI's private `_build_fresh_agent` (`cli.py`) since reaching into another entrypoint module's underscore-prefixed internals would be a layering violation. (The former case-set runner, `run_case_set`/isolated-temp-workspace execution, was removed along with the JSON case-file feature ‚Äî see the note below.)
- **`results.py`** ‚Äî `EvalRunResult` (schema version, run id, timestamp, agent version, model, dataset ref, per-case `EvalScore`s, aggregate metrics); `write_run_result` writes atomically (`tempfile.mkstemp` + `os.replace`) under `.my_coding_agent/evals/<run_id>/result.json`; `load_run_result` round-trips it back, ignoring unknown keys so a reader written against an older schema version tolerates a newer record. After the atomic write, `_write_verdict_artifacts` (issue #193) writes a `verdict.json` ‚Äî `{run_id, case_id, passed, metrics, detail, result_path}` ‚Äî into `.my_coding_agent/<session_id>/` for each score naming a session whose directory still exists on disk (real-cwd config-driven runs; a write failure logs a warning and never fails the run ‚Äî same fault-tolerance posture as `tool_execution`'s `_write_artifact_file`; trace files (`events.jsonl`, `session_data.json`) are never touched.
- **`reporting.py`** ‚Äî `render_verdict(result)` (issue #193): the one shared terminal renderer for a completed run ‚Äî per-check status/id/metrics then wrapped `detail`/rationale (judge verdicts' `criteria`/`overall_rationale` get dedicated formatting, other `detail` shapes fall back to generic key/value lines), then the run-level summary line. `cli.py`'s `--config` handling on the main `my-coding-agent` entrypoint calls it after `write_run_result`.
- **`cli.py`** ‚Äî `my-coding-agent-eval` console script (Click): a `click.group()` hosting only the `compare` subcommand below (no bare invocation). A run is always produced elsewhere ‚Äî via `uv run my-coding-agent --config <run.yaml>` (`src/my_coding_agent/cli.py`, folded in when the `run` subcommand was removed) ‚Äî so this entrypoint's job is purely gating two persisted results against each other.
- **`compare.py`** ‚Äî `evals/compare.py` (issue #142): two-run regression gate consuming persisted #139 result records ‚Äî reads only what's already on disk, no re-run. `compare_runs(baseline, candidate)` computes per-metric aggregate deltas plus the per-case pass‚Üîfail flip set (`CaseFlip`, with `is_regression` true only for pass‚Üífail), so a flat aggregate can't hide a subset regression. It uses the exact string already carried on `EvalRunResult.dataset` (`"config:<path>"` for a config-driven run) as the comparability guard: a mismatch raises `DatasetVersionMismatchError` by default, or is downgraded to a printed warning via `allow_version_mismatch=True` ‚Äî a metric move against a changed config is not a real signal, so refusing is the default rather than the override. `ThresholdConfig` (per-metric floors + a `no_case_regressed` rule, default on) and `evaluate_verdict` turn the comparison into a `Verdict(passed, reasons)`, with a reason string naming every violated floor and every regressed case id ‚Äî never a silent failure. `cli.py`'s `compare` subcommand (`my-coding-agent-eval compare A B`, args each either a run id under `.my_coding_agent/evals/` or a direct result-directory path) wires this into a CI gate: prints deltas/flips/verdict, exits `0`/non-zero mirroring `my-coding-agent-webui --check`, advisory to a human merge decision and never an auto-merge.
- **`trajectory.py`** ‚Äî `TrajectoryScorer` (issue #140): scores a run's execution *path*, not just its final answer, registered under the `"trajectory"` ref. `load_trajectory(session_dir)` reuses `viewer.reader.load_session` for the parsed trace and reads `session_data.json`'s `tool_records` + cost rollup (`CostRollup`) ‚Äî the same source `RunResult.session_dir` points at. Four deterministic dimension functions each return a `DimensionScore` (`value` in `[0, 1]` + structured `detail`): `score_tool_selection` (flags `not_found` ‚Äî off-toolset ‚Äî outcomes, keyed off the literal `"not found"`/`"wrong arguments for"` substrings `engine/tool_execution`'s `invoke_tool` always embeds in a failure's `error` text, since the reason isn't persisted as its own field), `score_argument_validity` (flags `wrong_args` outcomes), `score_error_handling` (replays `pipeline.anomaly.trailing_streak` over successive prefixes of `tool_records` ‚Äî the same signal `AnomalyDetectNode` computes live, one step at a time ‚Äî to catch a same-signature failure streak the run never reacted to, even if a later call broke it), and `score_efficiency` (steps/tokens/wall-clock from the cost rollup, scored against an optional case-supplied `EfficiencyBaseline`, plus a baseline-free `score_redundancy` sub-signal: work performed after the last call to a state-changing tool ‚Äî `STATE_CHANGING_TOOLS = {"bash", "write_file"}`, a conservative allowlist rather than trying to classify individual `bash` commands). `TrajectoryScorer.score` averages the four dimension values into one `EvalScore`, passing at/above `case.expected["pass_threshold"]` (default `0.7`).
- **`judge.py`** ‚Äî rubric-based LLM judge (issue #103), the only non-deterministic scorer. A `Rubric` (declared criteria, a shared integer score scale, per-criterion anchors) is either a disk JSON artifact loaded by `load_rubric` or an inline mapping of the same shape declared directly in `case.expected["rubric"]` (issue #200) ‚Äî `resolve_rubric` branches on the value's type (a mapping triggers the inline path, anything else resolves as a path) and both forms validate through the same `_rubric_from_dict`; a case never grades against a free-form "is this good?" prompt, and a malformed rubric (inline or on disk) raises `RubricError` rather than falling back to one. `score_with_judge` makes one bounded model call through `engine.llm` tagged with the distinct `CALL_KIND_JUDGE` call kind (so judge cost is captured and attributable like any other call) and parses the response into a `JudgeVerdict` ‚Äî a score + rationale per criterion plus an overall score/rationale, so systematic judge bias (verbosity/confidence preference) is inspectable in the persisted result rather than hidden behind one number. Response parsing first strips `<think>...</think>` reasoning, then extracts the JSON candidate ‚Äî the last fenced ` ```json ` block if present (the model's actual answer, which follows any illustrative example it may have echoed earlier in its own reasoning), else the outermost `{...}` span, else `JudgeError` ‚Äî and, before any of that, checks `extract_finish_reason` (`engine/llm/parsing.py`): a `finish_reason == "length"` (truncated) response raises `JudgeError` immediately rather than risking a partial or example snippet being mistaken for the real verdict. `JudgeScorer` adapts a `JudgeVerdict` to the `scoring.Scorer` contract and self-registers under the `"judge"` ref on import; an errored agent run, missing/malformed rubric, truncated response, or unparseable verdict are all recorded as a failed `EvalScore` with the reason in `detail`, never a crashed run. `calibrate` scores a human-labelled set with the judge and reports agreement via `cohens_kappa`, flagging the judge `reliable=False` when agreement falls below `DEFAULT_RELIABILITY_THRESHOLD` rather than letting an uncalibrated judge be silently trusted.
- **`run_config_file.py`** ‚Äî declarative YAML run config (issue #192): a single plain-text YAML file as the complete, self-contained definition of an eval/pipeline run (LLM connection, prompts, run parameters, inline evaluation checks ‚Äî an example lives at `examples/eval_run_config.yaml`), so it can be read/diffed/edited without a running server or the web UI. `load_config_file` parses with `yaml.safe_load` and validates before anything runs ‚Äî malformed YAML, an unknown top-level/section key, a missing `run.task`, an empty or invalid `evaluation.checks`, an unknown evaluator ref, or a raw `llm.api_key` value (rejected in favor of `api_key_env`, an environment-variable *name*, never a secret value) are collected and raised together as one `ConfigValidationError`. Connection fields resolve in a fixed order ‚Äî config value, then environment variable, then documented default ‚Äî with "config" as the source label in place of the interactive path's "saved settings". The parsed `llm`/`run` sections map onto the existing `RunConfig` model and `evaluation.checks` onto the existing `Check` model ‚Äî no parallel schema. `execute_from_config` reuses existing machinery only: the config's `task` runs as one real-cwd agent turn (mirroring `evaluation.run_evaluation`, so the session lands under the project's `.my_coding_agent/`, visible to the Traces tab), scored against every inline check via the scorer registry, aggregated into one `EvalRunResult` stamped with the config file's path and a SHA-256 content hash (`EvalRunResult.config_path`/`config_hash`, additive fields) so a result stays traceable to the exact configuration version that produced it. The main entrypoint's `--config <file>` flag (`uv run my-coding-agent --config <file>`, `src/my_coding_agent/cli.py`) wires this in: a validation failure exits `2` with every problem named, before any agent run or session directory is created; a scored run exits `0` on pass, `1` on fail. `_score_checks` stamps each `EvalScore` with `run_result.session_id` (mirroring `evaluation.py`'s identical `replace(...)` call) so `results._write_verdict_artifacts` (issue #196) can locate the session directory and write its `verdict.json` sibling ‚Äî without it, a config-driven run's session would never carry a discoverable verdict (session-verdict-visibility, issue #198).

**JSON case-file feature removed.** The former `cases.py` (`load_case_set`/`save_case`/`delete_case`, one `EvalCase` per `*.json` file under `.my_coding_agent/evals/cases/`) and `datasets.py` (versioned case-id collections under `.my_coding_agent/evals/datasets/<dataset_id>/versions.jsonl`, issue #141) were deleted, along with `runner.py`'s `run_case_set`/`_run_and_score_case` (isolated-`tempfile.TemporaryDirectory` per-case execution) and `cli.py`'s bare `my-coding-agent-eval`/`--cases` invocation ‚Äî every eval run is now defined declaratively via YAML (`run_config_file.py`) rather than as a directory of loose case files. `EvalCase` itself stays (`schema.py`): `run_config_file.py` constructs one internally per config so scoring (`scoring.py`/`judge.py`/`trajectory.py`) has a single shared type to score against. `.my_coding_agent/evals/rubrics/cve_aggregation_rubric.json` (un-ignored via the same `.gitignore` negation pattern as `.my_coding_agent/skills/`) stays live: the `cve_aggregation` example config's judge check still references it by path for delegation, source correctness, and synthesis quality scoring against live external APIs (CISA KEV via `vulnerability.circl.lu`, GitHub Advisories via `api.github.com`) rather than recorded fixtures ‚Äî deliberate, since a fixed expected answer would defeat the point of judge-scoring it.

## Evaluation entity: RunConfig + EvalConfig (issue #162)

An evaluation used to live only in the operator's head: nothing bound *how
the agent runs* to *what its output is checked against*, so it couldn't be
saved and re-run as one unit ‚Äî the #141/#154 dataset/case system versions
case membership, but a case's task and expected payload are still edited by
hand per case, with no reusable "run recipe." `evals/evaluation.py` (leaf
module, no dependency on `engine`/`pipeline` beyond driving `AgentNode` at
run time) adds three frozen-dataclass domain models, one JSON file per
object under `.my_coding_agent/evals/{run_configs,eval_configs,evaluations}/`
(`_write`/`_read`/`_list`, a one-file-per-object convention):

- **`RunConfig`** ‚Äî a persisted, reusable "how the agent runs": prompt
  template(s), model/provider, temperature/tokens/top_p, tool/memory/
  retrieval config, env vars. `_run_config_task` joins
  `user_prompt_template` + `context_template` (falling back to
  `description`/`name`) into the task string the agent actually runs.
  Standalone from any specific check ‚Äî the same RunConfig can back
  multiple Evaluations.
- **`EvalConfig`** ‚Äî a persisted, reusable ordered `Rule -> Check` tree.
  Each `Check` (id, name, method label, evaluator ref, `expected`,
  `threshold`) maps onto the exact same `expected` shape `EvalCase.expected`
  already carries per scorer (`_check_expected`
  translates a Check's `expected`/`method`/`threshold` onto the scorer
  registry's existing contract ‚Äî no new scoring semantics). `EvalConfig`
  creation/update calls `validate_eval_config`, which resolves every
  Check's `evaluator` against `scoring.resolve_scorer` up front
  (`UnknownEvaluatorError` on an unregistered ref) rather than failing
  silently at run time.
- **`Evaluation`** ‚Äî the first-class binding: `run_config_id` +
  `eval_config_id` + a `last_run` summary (`run_id`, `verdict`, `at`).
  `_validate_references` checks both ids resolve to a stored file before
  a create/update is written, so an Evaluation can never point at a
  missing RunConfig/EvalConfig. This is the only one of the three objects
  the (since-retired, issue #194) web UI's Evaluations tab listed and
  rendered ‚Äî a RunConfig or EvalConfig with no Evaluation referencing it
  was invisible there, even though it still existed on disk.

`run_evaluation` drives one agent turn (`_run_agent`) per the Evaluation's
RunConfig, then scores every Check across every Rule in its EvalConfig
against that single run's output (`_score_checks`, via the existing
scorer registry) and writes the result through the existing
`results.write_run_result` ‚Äî this module adds no new execution or scoring
engine, only the binding, persistence, and orchestration on top of what
`evals/runner.py` and `evals/scoring.py` already do.
`run_evaluation` runs in the process's actual current working directory ‚Äî
deliberately not isolated. Two things broke under isolation and drove this
choice: `AgentNode`'s session directory always resolves off `Path.cwd()`
(`pipeline/nodes/agent.py`), so a chdir'd-into-a-tempdir run wrote its session
there and lost it when the tempdir was cleaned up, meaning an Evaluation's
run never appeared in the Traces tab; and `_score_checks` ran before the
chdir was undone, so a judge Check's relative rubric path
(`case.expected["rubric"]`) resolved against the tempdir and failed with
"rubric file not found" even when the file existed at the real project
path. Running in the real cwd fixes both ‚Äî the session lands under the
project's actual `.my_coding_agent/` (visible to Traces) and relative
paths resolve where the operator expects.

The HTTP API that drove `run_evaluation` from the browser (`webui/evals/api.py`,
its CRUD/run routes under `/api/evals/{evaluations,run-configs,eval-configs}`)
and the embedded two-pane Evaluations page (`webui/evals/server.py`) were
removed with the rest of the web UI's Evals and Admin tabs (issue #194); the
